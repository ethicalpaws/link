## 1. 命名与定位

### 1.1 全称

**LINK — Long-term Intelligent & Individual Navigation Keystone**

### 1.2 简称

**LINK**

"LINK" 这个词，精准地概括了它要做的事情：**连接**现在与未来，连接我与我想成为的人，连接我与那些需要同样陪伴的人。

> **帮你走得更远。**

这句话是 LINK 所有设计的最终检验标准。无论 LINK 如何生长，它都必须回答同一个问题：这是否在帮助人们走得更远？

## 2. 缘起：从真实经历中生长出来

### 2.1 一个无法忽视的错位感

我对大学系统的感受，经历了从"失望"到"看清"的过程：

| 我看到的是什么 | 我真正理解的是什么 |
|:---|:---|
| "培养人才" | 统一管理大量学生 |
| "检验学习成果" | 产生可排序的数据 |
| "确保到课率" | 确保学生在流程中 |
| "实践教学" | 按规范完成可评分的过程 |

当我把这些安排放在"成长"的逻辑下审视时，它们显得荒谬。但当我意识到它们的真实逻辑是"管理"时，一切都说得通了。这个认知本身就是 LINK 诞生的土壤——**我意识到了现有系统并不是在支持我的方向，然后我决定自己建造一个系统来做这件事。**

### 2.2 演化路径

```
知识库 → 修桥工程 → LINK
```

每一步，都是对"我真正需要什么"的回答深化。每一步，也都拓展了它可能服务的人群。

### 2.3 情感起点

这个项目的起点，不仅仅是一个理性的判断。在知识库阶段，我独自学习了四周，感受到了真实的孤独。我需要一座桥，来让自己成为那个值得被看见的人。

这座桥已经长出了比最初动机更宏大的形状。

## 3. 核心理念：它与其他项目不同在哪里

| 项目类型 | LINK 的差异化 |
|:---|:---|
| AI 编程助手 | LINK 不生成代码，它管理成长 |

LINK 的独特之处不在于技术有多前沿，而在于它回答了一个被长期忽视的问题：**如何用 AI 支持一个人在数年甚至数十年的时间里，持续走向自己想去的方向。**

## 4. 五大支柱：LINK 的设计框架

LINK 的设计围绕一个核心问题展开：**如何帮助一个人走远路？**

我提炼出五个关键支柱，LINK 的所有机制，都是为了让这五个支柱在用户的日常中持续运转。

### 4.1 🧠 记忆 — 不遗忘

**问题：** 人会遗忘。昨天想通的事，今天又绕回去了。上个月学的东西，下个月就只剩模糊印象。

**LINK 的做法：**

```
原始消息（500 条上限）
  → 日摘要（每天 23:55，≥3 条则 LLM 生成，保留 14 天）
    → 周摘要（每周日 23:57，汇总 7 天，保留 52 周）
      → 归档（永久保留）
```

每次对话时，系统自动将近期摘要、核心身份、阶段计划注入到 Agent 的上下文中，让 Agent 知道你走到哪了。

**技术实现：** `packages/memory/` — storage.js、pipeline.js、context.js

### 4.2 🧭 导航 — 不迷失

**问题：** 知道要去哪，但每天被琐事推着走，走着走着就偏了。

**LINK 的做法：**

四层计划体系，覆盖不同时间尺度：

| 计划类型 | 周期 | 作用 |
|:---|:---|:---|
| 阶段计划（period） | 季度/半年 | 长期方向，大目标 |
| 月计划（monthly） | 月度 | 中期拆解 |
| 周计划（weekly） | 周度 | 当下执行 |
| 日计划（隐式） | 每天 | Agent 根据上下文主动提醒 |

Agent 在每次对话中都能读到这些计划，主动帮你校准方向："你本周的目标是 X，现在进度如何？"

**技术实现：** `routes/memory.js` — `/api/memory/plan/:type`，管理面板计划页面

### 4.3 🔍 检验 — 不自欺

**问题：** 学没学会，只有自己知道。但人很容易骗自己——"我看过了 = 我会了"。

**LINK 的做法：**

Agent 通过搜索知识库来判断用户的学习进度：

- **知识库双向搜索**：Agent 搜索用户笔记 + 外部知识库（技术用户可指向 Obsidian/VS Code 目录）
- **主动提问**：根据笔记内容，Agent 会主动检验理解："我看到你记了事件循环的笔记，你理解微任务和宏任务的执行顺序了吗？"
- **里程碑记录**：手动记录关键节点，Agent 据此判断阶段进展

**技术实现：** `packages/knowledge/` + `search_knowledge` 工具 + Agent prompt 构建

### 4.4 🏺 沉淀 — 不白费

**问题：** 经历过了，但没留下什么。时间过去了，人没有变厚。

**LINK 的做法：**

每一次对话、每一次工具调用、每一次权限拒绝，都在沉淀：

| 沉淀形式 | 内容 | 流向 |
|:---|:---|:---|
| 日摘要 | 每天的关键讨论和技术进展 | 注入上下文 |
| 周报 | 每周的趋势和关键决策 | 注入上下文 |
| 里程碑 | 用户手动标记的重要节点 | 成长轨迹 |
| 审计日志 | 工具调用记录和权限事件 | 开发者自查 |
| 知识库笔记 | 用户的学习记录 | Agent 搜索 |

**技术实现：** 摘要管道 + 里程碑 + 审计日志（`audit.js` + `data/audit.log`）

### 4.5 ⚙️ 定制 — 不僵硬

**问题：** 每个人的方向不同、习惯不同、技术能力不同。一套固定流程覆盖不了所有人。

**LINK 的做法：**

多层定制体系，从零代码到全自定义：

| 定制层 | 面向用户 | 做什么 |
|:---|:---|:---|
| **YAML 工具** | 所有人 | 写 YAML 就能加工具，零代码 |
| **战友配置** | 所有人 | 管理面板配置权限、角色、身份 |
| **Skill 编排** | 进阶用户 | 多步骤工作流，编排工具组合 |
| **MCP 协议** | 开发者 | 连接标准 MCP Server，自动发现工具 |
| **社区包** | 开发者 | 发 npm 包扩展 LINK 功能 |
| **外部知识库** | 技术用户 | 用自己的编辑器维护笔记，LINK 只读搜索 |
| **内置知识库** | 非技术用户 | 管理面板直接写笔记 |

**技术实现：** `packages/registry/` — ToolRegistry、SkillRegistry、MCPBridge、YamlLoader

### 5.3 核心数据流

```
用户输入 → POST /api/chat/stream
  → @提及解析 → 确定回应者
  → 并行调用多个 Agent
    → callWithTools()
      → 权限白名单过滤
      → 工具指令注入
      → LLM 调用（180s 超时）
      → 有 tool_calls?
        → 中间件管道：
          权限检查 → 参数校验 → 路径安全
          → 执行（指数退避重试 1s→2s→4s）
          → 结果截断 → 审计日志
        → 再调 LLM（最多 5 轮）
      → 无 tool_calls? → 返回文本
  → SSE 推回前端
  → 房间历史持久化

定时任务（每 30s tick）：
  → 用户定时任务 → LLM 执行 → 推送到房间
  → 23:55 日摘要 → 周日 23:57 周报 → 23:58 归档
```

### 5.4 安全架构

| 攻击面 | 防护 |
|:---|:---|
| **命令注入** | command.js: 单引号转义；ssh.js: spawn + 参数数组 |
| **SSRF** | http.js: DNS 解析 + IP 黑名单 |
| **路径遍历** | 白名单校验 + isAgentDir() 检查 |
| **路径黑名单** | /etc /sys /proc /C:\Windows 等阻断 |
| **XSS** | Vue 自动转义 + h() 兜底 |
| **工具权限** | 白名单模式，不配置 = 没权限 |
| **认证** | Basic Auth（config → 环境变量 → 随机密码） |
| **审计** | 所有工具调用记录到 data/audit.log |

### 5.5 外部依赖

| 包 | 用途 |
|:---|:---|
| express ^4.22 | HTTP 服务器 |
| marked ^18 | Markdown 渲染 |
| js-yaml ^4.1 | YAML 解析 |
| gray-matter ^4.0 | Markdown frontmatter |
| @modelcontextprotocol/sdk | MCP 协议桥接 |

**仅 5 个 npm 依赖**，无数据库、无构建工具链、单进程、文件系统持久化。

### 5.6 技术理念

> **用最少的外部依赖，做最实质的事情。**

LINK 的技术选型遵循几条原则：

1. **不引入不必要的抽象** — 工具调用 200 行代码自己写，不引入 LangChain
2. **安全内置在每一层** — 不是加一个安全网关，是每个函数都有安全考量
3. **可审计** — 所有操作都可回溯，不黑盒
4. **无数据库** — 文件系统持久化，备份就是复制文件夹
5. **单进程可运行** — `npm start` 就跑起来，不依赖 Docker/K8s

## 6. 战友系统：多角色支撑体系

一个人走远路，需要的不只是一个工具，而是不同角色在不同时刻的支撑。

### 6.1 角色设计

战友角色完全由用户自定义，LINK 不预设任何角色。用户通过管理面板新增战友时，可以自由定义：

- **ID / 名称 / Emoji** — 身份标识
- **prompt** — 角色定位和行为描述
- **autoReply** — 是否在群聊中自动回应
- **tools** — 允许使用的工具（白名单）
- **skills** — 允许使用的 Skill
- **SOUL.md / AGENTS.md** — 深度身份定义文件

每个人的 LINK 中的战友阵容都是独特的，反映用户自己的需求和偏好。

### 6.2 身份定义

每个战友通过四个文件定义自己的身份：

```
agents/{id}/
├── SOUL.md     ← 灵魂底色、核心信条
├── AGENTS.md   ← 工作规范、职责、权限
├── TOOLS.md    ← 可用工具列表（自动生成）
└── USER.md     ← 用户信息（自动同步）
```

### 6.3 权限隔离

每个战友可独立配置工具权限和 Skill 权限：

```json
{
  "id": "coach",
  "tools": ["read_file", "write_file", "search_knowledge"],
  "skills": ["daily_news"]
}
```

不配置 = 没有任何权限。白名单模式，拒绝一切未经授权的操作。

## 7. 用户路径：从 onboarding 到长期陪伴

### 7.1 第一天

1. `npm install && npm start` — 浏览器打开 http://localhost:3000
2. 看到 💬 聊天界面，发送第一条消息
3. 看到管理面板，配置 LLM 模型和 API Key
4. 看到侧边栏的 🤖 战友、📚 知识库、📋 计划

### 7.2 第一周

1. 在知识库里写下第一篇笔记
2. 在计划里设定阶段目标
3. 在身份页面写下核心初衷
4. 在定时任务里设置每日回顾

### 7.3 第一个月

1. 日摘要自动生成，Agent 能记住上个月的事
2. 调整战友权限，限制某些工具的访问
3. 添加外部知识库目录（技术用户）
4. 配置 MCP Server，扩展工具生态

### 7.4 持续使用

1. LINK 成为日常习惯 — 每天至少和战友聊几句
2. Agent 能主动提醒目标进度、检验知识掌握
3. 里程碑积累，成长轨迹可视化
4. 定时任务自动跑日/周/月复盘

## 8. 未来愿景：它可能长成什么样子

### 8.1 一个新教育范式的种子

| 传统教育的逻辑 | LINK 可能提供的逻辑 |

### 8.3 社区生态

- 用户共享战友角色（SOUL.md + AGENTS.md）
- 社区贡献 Skill 包（通过 npm 发布 `@link/skill-*`）
